# Development Rules

## Core Principles

- Keep storage, CPU, and memory usage minimal.
- Keep code straightforward. Do not add abstraction unless it removes real duplication now.
- Breaking changes are allowed while product is young.
- UI can be plain. Correct behavior matters more than styling.
- No VRAM limit feature. Show GPU capacity/allocation only.
- Environments define available Jupyter Docker images. Admins manage them; users choose from enabled environments.
- Docker image reference is the runtime source of truth. Admins define environment packages; the platform validates that model before save and should own generated build details.
- Users request access. Admin approval grants permission. Users manually start workspaces from approved grants.
- Stop is destructive. Workspace data is wiped with the container/service.
- Do not expose time-window UX to users. Current hour fields exist only for compatibility until idle-activity cleanup replaces expiry internals.
- If no GPU node is available, show clean no-capacity message. Do not create a queue for MVP.
- Errors must be fixed directly. Do not hide, dodge, or wrap errors into vague messages.
- Do not use nested `try/catch`. Let errors fail clearly or handle once at the boundary.
- No realtime platform features for MVP. Use polling where needed.
- Workspace provisioning must be asynchronous. HTTP handlers enqueue/update state, then return quickly; scheduler/reconciler provisions Jupyter/Swarm/Caddy work in the background while UI polls status.

## Architecture

- Use a NestJS modular monolith.
- Use Nunjucks templates rendered by NestJS.
- Use PostgreSQL only. No SQLite fallback.
- Use Drizzle migrations committed to the repo.
- Every schema migration must be generated by Drizzle tooling, normally `pnpm db:generate`, from `src/infrastructure/db/schema.ts` changes. Do not write migration SQL manually.
- If a data cleanup/backfill is required, keep schema DDL generated by Drizzle and add the smallest hand-written DML only after documenting why Drizzle cannot generate it.
- After any migration change, verify `pnpm db:generate` is a no-op, `pnpm exec drizzle-kit check` passes, and a fresh isolated database can run `pnpm db:migrate`.
- Use Docker Swarm as the only orchestration path in local and production.
- Local development must run real single-node Docker Swarm.
- Do not create fake backend/provider implementations for orchestration.
- Do not hardcode nodes, GPU info, GPU counts, station names, or environment options to satisfy minimum requirements.
- Node/GPU inventory must come from Docker Swarm node labels/API plus health probes.
- Environments must come from PostgreSQL records managed by the platform UI or migrations, not hardcoded controller/service arrays.

## Module Boundaries

- Controllers stay thin: parse request, call service, render/redirect.
- Business logic lives in services.
- Docker Swarm calls live only in `SwarmModule`.
- Caddy API calls live only in `ProxyModule`.
- Database schema lives in `src/infrastructure/db/schema.ts`.
- API/routes must be clear and obviously used. Remove dead routes instead of keeping placeholders.
- If data is environment-specific, model it as DB/config/Swarm label data. Do not bury it in code.

## Testing

- Backend/domain tests are integration-focused.
- Tests should use local PostgreSQL and local Swarm when they touch persistence or orchestration.
- Full usability verification must include `pnpm smoke:local`, `pnpm e2e:local`, and browser-driven checks for changed pages.
- Fresh database verification must run migrations and seed against an isolated DB before setup docs are considered valid.

## Code Quality

- TypeScript uses balanced strictness.
- Avoid `any` and unsafe casts.
- Use `@/` alias for cross-module imports. Avoid deep `../../..` imports.
- Use `class-validator` DTOs for forms/API input.
- Use Prettier + ESLint.
- Keep frontend JavaScript minimal.

## Operations

- Use `.env` plus validation for config.
- Do not commit secrets.
- In-process scheduler handles reconciliation and cleanup for MVP.
- Do not provision Jupyter workspaces synchronously inside user-facing HTTP request handlers.
- Audit/logging starts basic; expand only when necessary.

## Local Dev Rule

Every developer should be able to test full access request -> approval grant -> manual workspace start -> Swarm service -> proxy -> destructive stop lifecycle locally without connecting to the real lab cluster.
